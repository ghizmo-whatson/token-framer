# token-framer (Image + Frame Mixer)

Браузерный инструмент для «вставки» картинки в декоративную рамку (token frame) с
возможностью добиться эффекта **«вылезания» изображения из токена** — когда часть
картинки (голова, оружие, лапа персонажа и т.п.) выходит поверх рамки, создавая
иллюзию объёма / выхода за пределы окна.

Проект — это **обычный статический одностраничник** (plain HTML + CSS + vanilla JS,
без сборки и зависимостей). Хостится на GitHub Pages и открывается прямо там же.
Весь рендеринг и обработка изображений делаются на клиенте через `<canvas>`; ничего
на сервер не отправляется.

- Репозиторий: https://github.com/atbakalov95/token-framer
- Язык интерфейса: русский по умолчанию, есть переключатель RU/ENG (см. i18n ниже).

---

## Что значит «токен» и «рамка»

Терминология взята из настольных/VTT игр (Roll20, Foundry и пр.):

- **Токен (token)** — картинка персонажа/существа (левая колонка). В коде — `token`.
- **Рамка (frame)** — декоративная оправа с прозрачным или светлым «окном» в центре,
  куда вставляется токен (правая колонка). В коде — `frame`.
- **Результат (composed)** — итоговая композиция: токен, видимый сквозь окно рамки,
  рамка поверх него, и (в режиме «Маска») части токена, нарисованные поверх рамки —
  это и есть эффект «вылезания» (центральная колонка).

Папка [ai_generated/](ai_generated/) содержит **пресеты рамок** (декоративные оправы с
прозрачным центром, сгенерированы нейросетью) и подключена к приложению через кнопку
«Пресеты» в правой колонке (см. раздел «Пресеты рамок» ниже). Папка
[frame_references/](frame_references/) — локальные референсы, кодом не используется.
Помимо пресетов изображения можно грузить вручную через drag-and-drop или буфер обмена.

## Раскладка интерфейса (3 колонки)

```
┌──────────────┬─────────────────────┬──────────────┐
│  ТОКЕН        │       ЦЕНТР          │   РАМКА       │
│ (левая)       │   (результат)        │  (правая)     │
├──────────────┼─────────────────────┼──────────────┤
│ Вставить /    │ Режим: Смещение/     │ Вставить /    │
│ Откатить      │        Маска         │ Откатить      │
│ ☐ Кисть       │ Масштаб холста %     │ ☐ Кисть       │
│   удаления    │ Масштаб токена %     │   удаления    │
│ [кисти]       │ [кисть маски]        │ [кисти]       │
│               │                      │               │
│  canvas       │   canvas (preview)   │   canvas      │
│  (токен)      │   + кнопка Скачать   │   (рамка)     │
└──────────────┴─────────────────────┴──────────────┘
```

Левая и правая колонки идентичны по функционалу (одна для токена, другая для рамки) —
обе создаются одной фабрикой `createSideController(side)`. Центральная колонка
собирает финальное изображение и управляет режимами.

## Архитектура кода ([main.js](main.js))

Весь код обёрнут в один IIFE. Глобального экспорта нет.

### Главный объект состояния — `state`
- `language`, `appMode` (`offset`/`mask`), `scalePercent`, `tokenScalePercent`,
  `pendingPasteTarget`.
- `state.sides.token` и `state.sides.frame` — симметричные структуры на сторону:
  - `source` — исходный `<canvas>` загруженного изображения (canvas, не Image).
  - `masks: { color, local, eraser }` — три **отдельных canvas-маски удаления**.
    Каждая «вычитается» из source (см. `composeEffectiveCanvas`).
  - `enabled`, `brushMode` (`color`/`local`/`eraser`), `brushSize`, `localBrushSize`.
  - `viewport` — `zoom`/`panX`/`panY` (только для просмотра в панели; на итог не влияет),
    `displayRect`/`displayScale` — где и в каком масштабе нарисован canvas, нужно для
    пересчёта координат мыши в пиксели изображения.
  - `paint` — текущее состояние взаимодействия (рисуем / тянем / последние координаты).
  - только у `frame`: `assets: { overlay, windowMask }` — производные ассеты (см. ниже).
- `state.center` — оффсет токена (`baseOffsetX/Y`), `userMask` (canvas маски «вылезания»),
  `composed` (последний собранный итоговый canvas), zoom/pan превью, размер кисти маски,
  флаги перетаскивания/рисования, `displayRect`/`displayScale`.

### Ключевые сущности
- `ui = bindUI()` — словарь всех DOM-узлов (token / frame / center / app).
- `sideControllers.token` / `.frame` — контроллеры боковых панелей, каждый возвращает
  `{ setSource, renderPreview, resetMasks, rebuildAssets }`.
- `createSideController(side)` — замыкание со всеми обработчиками панели (drop, кисти,
  pan/zoom, undo, paste) и приватными функциями рендера/штрихов.
- `setupAppEvents()` — события центра и приложения (режим, масштабы, кисть маски,
  мышь центра, paste, скачивание, ресайз).

## Пайплайн композиции (сердце «вылезания»)

Ключевые функции: `composeFinalImage()`, `buildFrameAssets()`, `computeTokenPlacement()`.

### 1. «Эффективные» изображения — `composeEffectiveCanvas(source, color, local, eraser)`
Берёт source и через `globalCompositeOperation = "destination-out"` «вычитает» из него
все три маски удаления. Результат — токен/рамка с уже удалёнными цветами и стёртыми
участками. Используется и для превью панели, и для финальной сборки.

### 2. Авто-определение «окна» рамки — `buildFrameAssets(frameCanvas)`
Чтобы токен можно было показать сквозь центр рамки, код сам находит прозрачное «окно»:
1. Помечает «проходимые» (`passable`) пиксели — прозрачные (`a < 32`), почти белые,
   или полупрозрачные яркие малонасыщенные (эвристика под белый/прозрачный фон окна).
2. Делает **заливку (flood fill, BFS)** от центра рамки (`findWindowSeed` ищет ближайший
   проходимый пиксель к центру) — получает связную внутреннюю область = окно.
3. Строит два ассета:
   - `windowMask` — белая маска формы окна (где токен должен быть виден).
   - `overlay` — копия рамки, но с «вырезанным» (прозрачным) окном → это арт рамки
     без центра, который кладётся поверх токена.

Ассеты пересобираются при загрузке рамки и после каждого штриха по рамке
(`rebuildDerivedAssets`).

### 3. Размещение токена — `computeTokenPlacement()`
Токен масштабируется так, чтобы **покрыть** окно рамки (`max` из отношений сторон),
затем умножается на `tokenScalePercent`, центрируется и сдвигается на `baseOffset`
(перетаскивание в режиме «Смещение»).

### 4. Сборка — `composeFinalImage()`
Размер холста = размер рамки (или токена, если рамки нет) × `scalePercent`. Шаги:
1. `maskedBase`: рисуем токен → `destination-in` с `windowMask` → токен виден только
   в окне; дополнительно ограничивается прямоугольником границ рамки.
2. Кладём `maskedBase` на итог.
3. Сверху кладём `overlay` (рамка с вырезанным окном) — рамка оказывается над токеном.
4. **Режим «Маска»**: если есть `state.center.userMask`, рисуем ещё один слой токена,
   обрезанный по нарисованной пользователем маске, и кладём его **поверх рамки**. Это
   и есть эффект «вылезания» — пользователь кистью указывает, какие части токена должны
   быть над рамкой.

Если рамка не загружена — просто рисуется токен по центру (рамка опциональна).

## Возможности и инструменты редактирования

### Загрузка изображений
- **Drag-and-drop** файла в левую/правую область (`setupDropArea`).
- **Кнопка «Вставить из буфера»** → `readClipboardImage()` (Async Clipboard API; фолбэк
  на `data:image/...` из текста). Если браузер не дал доступ по клику — выставляется
  `pendingPasteTarget` и ловится событие `paste` (Cmd/Ctrl+V) через `onWindowPaste`.

### Инструменты очистки на токене и рамке (чекбокс «Кисть удаления …»)
Три режима кисти (radio), у каждого свой эффект; **ЛКМ применяет, ПКМ отменяет/возвращает**:
1. **Удаление цвета** (`color`, фиолетовая кисть) — `applyGlobalColorMask`: собирает
   цвета под мазком и удаляет **все пиксели этого точного RGB по всему изображению**.
   Точное совпадение цвета (ключ `r<<16|g<<8|b`). Удобно убрать сплошной фон.
2. **Локальная очистка цвета** (`local`, фиолетовая + красная кисти) —
   `applyLocalColorMask`: красной кистью (`localBrushSize`) сэмплит цвета, удаляет их
   **только внутри области большой фиолетовой кисти** (`brushSize`) вдоль мазка.
3. **Ластик** (`eraser`) — `applyEraserMask`: обычное стирание мазком (freehand).

Каждый режим пишет в свою маску (`masks.color/local/eraser`). Итог = source минус все
три маски. **«Откатить изменения»** (`resetMasks`) сбрасывает все три маски стороны.

### Просмотр в боковых панелях (не влияет на итог)
- **Колесо мыши** — зум 1×–8× относительно курсора.
- **Средняя кнопка (или ЛКМ при выключенном чекбоксе)** — панорамирование.
- Кружок-курсор показывает размер активной кисти (две окружности в режиме `local`).

### Центральная колонка
- **Режим «Смещение» (`offset`)** — ЛКМ-перетаскивание двигает токен внутри рамки
  (`baseOffsetX/Y`).
- **Режим «Маска» (`mask`)** — появляется кисть; ЛКМ рисует маску «вылезания» (части
  токена поверх рамки), ПКМ стирает маску. Колесо — зум превью, средняя кнопка — пан.
- **Масштаб** (`scaleSelect`, 50–200%) — размер итогового холста (поля вокруг рамки).
  При изменении маска центра переразмеривается с сохранением центра
  (`resizeCenterMaskCanvas`).
- **Масштаб токена** (`tokenScaleSelect`, 50–200%) — размер токена в композиции.
- **Скачать** — `downloadComposedImage()` сохраняет `composed` как `mixed_image.png`
  (PNG через `canvas.toBlob`).

## Пресеты рамок (кнопка «Пресеты»)

Кнопка **«Пресеты»** вверху правой колонки открывает модальное окно с галереей готовых
рамок из папки [ai_generated/](ai_generated/). Превью-карточки показывают **и имя, и само
изображение** (на шахматном фоне, чтобы была видна прозрачность). Окно закрывается
крестиком, кликом по затемнению или клавишей Esc. Клик по карточке загружает выбранную
рамку в правое окно (`sideControllers.frame.setSource`) и закрывает галерею. В шапке окна
выводится пометка: **все пресеты сгенерированы нейросетью**.

### Как устроено перечисление файлов (без хардкода путей)
Цель — чтобы новые пресеты добавлялись **простым добавлением файлов в папку**, без правки
кода. Имена файлов нигде не хардкодятся; список читается динамически (`loadPresetList`),
два слоя по очереди:
1. **`listPresetsFromDirectoryIndex`** — `fetch("ai_generated/")` и парсинг HTML-листинга
   каталога (`DOMParser` → ссылки `<a href>`). Работает на дев-серверах с авто-индексом
   (напр. `python3 -m http.server`) — пресеты видны сразу, без коммита.
2. **`listPresetsFromGitHub`** — фолбэк через GitHub Contents API
   (`api.github.com/repos/{owner}/{repo}/contents/ai_generated`). Работает на GitHub
   Pages, где авто-индекса нет. `owner`/`repo` определяются из `*.github.io` хоста
   (`resolveGitHubRepo`), иначе берутся из `PRESETS.fallbackRepo`.

Оба слоя фильтруют файлы по расширению (`isImageName`, см. `PRESETS.imageExtensions`),
поэтому мусор вроде `.DS_Store` отсекается. Каждый пресет получает **два URL**
(`makePreset`): `localUrl` (`ai_generated/<имя>`, «рядом» со страницей) и `rawUrl`
(копия на `raw.githubusercontent.com`, строится из `resolveGitHubRepo`). Список
кэшируется (`PRESETS.cache`).

### Как грузится сама картинка (local-first, фолбэк на GitHub)
`applyPreset` пытается источники по очереди (`loadUsableCanvas`):
1. **`localUrl`** — локальная копия рядом со страницей (без CORS).
2. **`rawUrl`** — копия с GitHub (с `crossOrigin="anonymous"`), если локальная не нашлась
   или непригодна.

После загрузки canvas проверяется на «tainted» через пробный `getImageData(0,0,1,1)` в
`try/catch`: если canvas «испорчен» (типично для `file://`, где локальная картинка
портит canvas), источник отбрасывается и берётся следующий. Это важно — иначе позже
молча упали бы «Скачать» (`toBlob`) и инструменты цвета (`getImageData`). Превью-карточки
тоже сначала пробуют `localUrl`, по `error` переключаются на `rawUrl` (для показа taint
не важен).

### Поведение в разных окружениях
| Где открыто | Список пресетов | Картинка |
|-------------|-----------------|----------|
| **GitHub Pages (деплой)** | GitHub API | локально (same-origin), без CORS |
| **Локальный веб-сервер** (`python3 -m http.server`) | авто-индекс каталога | локально, без CORS |
| **Открытие `file://` напрямую** (двойной клик по HTML) | GitHub API (нужен пуш) | с GitHub (локальная «портит» canvas → фолбэк) |

То есть фича работает во всех трёх случаях, **но `file://` и деплой требуют, чтобы
картинки были закоммичены и запушены** (для списка через GitHub API и для `rawUrl`).
Чисто локально, **без пуша**, протестировать можно через локальный веб-сервер
(`python3 -m http.server` в корне проекта, затем открыть `http://localhost:8000`) —
там список берётся из авто-индекса, а картинки грузятся локально.

> Почему `file://` не находит пресеты без пуша: страница из `file://` не может
> «прочитать» содержимое папки (нет ни сервера с авто-индексом, ни API списка файлов),
> поэтому список приходит только из GitHub. Без коммита там пусто.

### Чтобы добавить новый пресет
1. Положить файл-изображение (png/jpg/webp/…) в папку `ai_generated/`.
2. **Закоммитить и запушить** в ветку, с которой деплоится GitHub Pages (обычно `main`).
   (Для предварительного локального теста достаточно дев-сервера — см. таблицу выше.)

Конфиг фичи — константа `PRESETS` в начале [main.js](main.js) (`dir`, `imageExtensions`,
`fallbackRepo`). Логика — `setupPresets` / `openPresetsModal` / `renderPresetCards` /
`applyPreset` / `loadUsableCanvas` / `makePreset` / `buildRawUrl` и функции перечисления.

## Рендер и производительность

- Превью боковых панелей: `renderPreview()` (внутри контроллера) — `fitRect` вписывает
  изображение в область, применяет zoom/pan, запоминает `displayRect`/`displayScale`.
- Превью центра: `renderCenterPreview()` → вызывает `composeFinalImage()` каждый кадр
  изменения. Композиция полностью пересобирается на каждый штрих/сдвиг (canvas-операции).
- `requestLayoutRender()` — двойной `requestAnimationFrame` для перерисовки после
  изменений layout (показ/скрытие контролов кистей).
- `getImageData`/`putImageData` с `willReadFrequently: true` для попиксельной обработки
  цветовых масок.
- Преобразование координат: `widgetToImagePoint()` переводит точку виджета в пиксель
  исходного изображения с учётом `displayRect`.

## Интернационализация (i18n)

- Словарь `I18N` с ключами `ru`/`en`, функция `t(key, vars)` с подстановкой `{size}` и
  фолбэком на русский.
- Переключатель языка вверху справа (`langSelect`). `applyLanguage()` проставляет тексты,
  подписи опций масштаба токена и **подсказки (`title`) ко всем контролам** (`applyTooltips`)
  — там же подробно по-русски/английски описано, что делает каждый элемент.
- Названия инструментов привязаны к цвету кисти: «Фиолетовая кисть» = основная,
  «Красная кисть» = сэмпл цвета в локальном режиме.

## Замечания / ограничения

- Маски рисуются в координатах **исходного** изображения, поэтому правки сохраняют
  полное разрешение независимо от зума панели.
- Зум/пан боковых панелей — чисто визуальные, в итог не попадают.
- Удаление цвета сравнивает RGB **точно** (без допуска) — для антиалиасинговых краёв
  может оставаться кайма; для таких случаев есть локальный режим и ластик.
- Эвристика поиска окна (`buildFrameAssets`) рассчитана на рамки с прозрачным/светлым
  центром; на нестандартных рамках окно может определиться неточно (тогда чистят рамку
  кистями вручную).
- Чистый клиент, без бэкенда и хранения: при перезагрузке страницы состояние теряется.
